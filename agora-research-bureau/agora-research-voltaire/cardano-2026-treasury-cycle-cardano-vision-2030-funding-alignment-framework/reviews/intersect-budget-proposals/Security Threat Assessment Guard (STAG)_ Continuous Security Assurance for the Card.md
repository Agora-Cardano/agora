# Alignment with Cardano Vision 2030 - Security Threat Assessment Guard (STAG): Continuous Security Assurance for the Cardano Ecosystem

## Proposal Under Review

**Security Threat Assessment Guard (STAG): Continuous Security Assurance for the Cardano Ecosystem**

* **Secondary Pillar:** None
* **Confidence:** High

## AI Rationale

The STAG proposal’s core objective is to rigorously test and secure Cardano’s core infrastructure. It is a continuous “red team” programme targeting vulnerabilities across the protocol stack, including UPLC smart contract execution, ledger rules, consensus, networking, wallets, compiler toolchains, CLI, and mainnet performance monitoring.

This directly supports Pillar 1: *Infrastructure & Research Excellence*, which aims to keep Cardano secure, fast, and interoperable. By contrast, the proposal does not primarily focus on building user-facing applications or industry-specific use cases, community engagement initiatives, or on-chain governance mechanisms. The fact that STAG is funded by the community, rather than IOG, the Cardano Foundation, or Intersect, is relevant to its independence but incidental to its core mission.

Therefore, the proposal’s primary purpose aligns most directly with strengthening Cardano’s security, resilience, protocol assurance, and operational infrastructure, fitting **Infrastructure & Research**.

## Supporting Evidence

### Evidence 1

The proposal emphasizes Cardano’s multi-client architecture and hidden protocol risks:

> “Cardano is now a multi-client blockchain. Amaru is enacted; Dingo and Gerolamo are in active development; Leios will introduce further implementation surface. Wherever two independent implementations handle a consensus edge case, a state transition, or a mini-protocol message differently, an exploitable inconsistency exists — invisible to any review examining a single node. That is the class of vulnerability STAG is built to find.”

This highlights that STAG targets core protocol security issues, which directly supports Pillar 1.

### Evidence 2

STAG is described as:

> “an independent, continuous red team programme for the full Cardano protocol stack: UPLC smart contract execution, compiler toolchains, ledger rules, consensus, peer-to-peer networking, wallet interfaces, and mainnet performance monitoring.”

This shows that the proposal’s focus is on the underlying infrastructure layers, aligning with Pillar 1’s goal of maintaining a secure and reliable base layer.

### Evidence 3

The proposal states that all critical findings are reported under responsible disclosure to the Security Council and relevant development teams, reinforcing that the programme is structured around protocol security assurance rather than adoption, marketing, or ecosystem growth.

### Evidence 4

The work packages cover UPLC and smart contract execution security, compiler toolchain security, ledger security and specification, consensus and network protocol security, CLI security, wallet and client security, and mainnet performance monitoring. These are all technical infrastructure domains.

## Classification Notes

### Primary Classification Justification

The proposal’s deliverables are dominated by technical security assurance, adversarial testing, protocol conformance, specification review, vulnerability discovery, responsible disclosure, security reporting, and operational monitoring.

This aligns most directly with **Pillar 1: Infrastructure & Research**, especially its focus on keeping Cardano secure, fast, interoperable, and capable of hosting more economic activity.

Other pillars are not selected as primary because they do not explain the proposal’s core purpose:

* **Adoption & Utility** may benefit indirectly if stronger security improves institutional confidence, but the proposal does not primarily deliver user-facing adoption, vertical-specific use cases, or direct utility growth.
* **Governance** is referenced through community funding, opt-out checkpoints, and Security Council reporting, but these are oversight and accountability mechanisms rather than the proposal’s main strategic output.
* **Community & Ecosystem** is not a primary fit because the proposal does not focus on talent acquisition, education, community growth, localized adoption, or market narrative.
* **Sustainability & Resilience** is relevant as an indirect effect because protocol security supports operational resilience, but the main work is technical security infrastructure rather than treasury management, tokenomics, SPO incentives, or financial stewardship.

For that reason, **Infrastructure & Research** is the strongest and most direct classification.
